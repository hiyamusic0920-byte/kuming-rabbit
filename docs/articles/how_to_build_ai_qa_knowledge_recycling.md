# 如何建立 AI QA 的知識回收機制？

上一篇我寫「如何讓 AI 理解需求」，核心是：不要讓 AI 一拿到 ticket 就開始寫測試，先讓它把自己的理解、假設和不確定攤開來，讓人可以校正。

但校正之後呢？

如果我今天花了十分鐘糾正 AI，告訴它某個欄位不能這樣解讀、某種狀態不能這樣測、某個 UI 行為以前踩過坑，但這些校正只留在當下那段對話裡，那下一個 session 重新開始時，它很可能又犯一模一樣的錯。

那一刻的感覺很像在不停地教同一堂課。

我後來才意識到，問題不是 AI 學不會，而是我沒有一個讓它「記得住、又不會堆爛」的機制。

所以這篇接著上一篇，只回答一件事：

> 怎麼讓一次校正不蒸發，而是沉澱成下次 AI 就懂的知識？

而且更重要的是，怎麼讓這些知識不要越堆越爛。

## 沒被回收的校正，就是白費

一開始我以為，教 AI 就是在對話裡糾正它。

它誤解需求，我補充脈絡。

它測錯方向，我告訴它真正風險在哪。

它選錯 selector，我貼一段正確做法。

這樣當下看起來有效，因為 AI 會立刻修正。但過幾天、換一個 session、換一個 agent，問題又回來了。

因為那次校正沒有流回任何持久的地方。

後來我把這件事重新看成一個迴圈：

```text
AI 出錯 / 誤解
  ↓
人類校正
  ↓
沉澱成一條可重用的知識
  ↓
回流到 AI 下次會讀到的脈絡
  ↓
下次 AI 不再從零開始
  ↺
定期修剪，避免知識越堆越爛
```

很多人做到第二步就停了：AI 錯了，人類改它。

但真正讓 AI QA 系統變強的，是第三步之後：校正有沒有變成一條可重用的知識？那條知識有沒有回到下次 AI 的脈絡？如果它過時或重複了，有沒有被修剪？

這也是我後來對 AI QA 的理解轉變。

重點不是「教 AI」。

重點是設計一個讓校正能流回去、而且越用越清楚的迴圈。

## 為什麼預設情況下，知識會蒸發

AI 很會在一段對話裡學你的偏好。

你糾正它幾次，它會在同一個 session 裡調整語氣、修正策略、避開剛剛踩過的坑。這很容易讓人誤以為它真的學會了。

但多數時候，它學到的是當下 context，不是長期知識。

當session 結束、context 被壓縮、換一個 agent 或換一條 thread 之後，這些校正就可能消失。留下來的通常只有人的記憶，以及一段很難被下一個 agent 自動讀到的聊天紀錄。

這就是第一個問題：人的記憶不是系統。

第二個問題是：prompt 也不是知識庫。

把所有規則塞進 prompt，看起來很直覺。今天多踩一個坑，就在 prompt 裡加一句；明天多發現一個限制，就再加一段。短期有效，但長期會變成另一種失控。

prompt 會越來越長，重點越來越模糊，互相矛盾的規則越來越多。AI 不是沒有讀到，而是讀到太多，抓不到哪一條才是現在最重要的。

所以我後來把「記得住」拆成兩件事：

1. 知識要有持久的家。
2. 知識要能被下次工作流程讀到。

只做到第一點不夠。你把文件寫得很完整，但 agent 從來不會在正確時機讀它，那還是等於不存在。

## 知識需要一條回家的路

在 my-playwright 裡，我沒有把所有知識都塞到同一個檔案。

因為 QA 知識不是同一種東西。

有些是產品規則，有些是測試資料，有些是 UI selector，有些是通用 QA 直覺，有些只是 agent 做事時要遵守的行為偏好。它們如果混在一起，短期好像方便，長期一定會爛。

所以我把知識分成幾個持久落點：

| 校正出來的知識 | 應該回到哪裡 |
|---|---|
| AI 該怎麼做事的行為規則 | memory |
| QA 該檢查什麼的通用直覺 | `docs/QA_HEURISTICS.md` |
| 特定產品或商業規則 | `docs/product/` |
| 測試資料、API 欄位、seed helper | `seeds/` |
| UI selector、等待條件、頁面互動陷阱 | `pages/_registry/` |

這篇不展開這些檔案各自怎麼設計。那是另一篇「AI QA 的治理與知識管理」要講的靜態架構。

這裡真正重要的是動態迴圈：當一次校正發生時，它有沒有一條路徑可以流進正確的家。

如果沒有這條路徑，再好的知識分類都只是空架子。

## 把一次校正變成一條規則

知識回收的最小單位，不是一整份文件，而是一條可以被下次引用的規則。

一條好的回收規則要夠小、夠明確、夠可判斷。

例如，不好的寫法是：

```text
測試時要注意資料。
```

這句太空。下次 AI 讀到，也不知道該怎麼改變行為。

比較好的寫法是：

```text
驗金額加總或計數前，先確認資料來源是否包含軟刪除或舊資料，避免把不應計入的資料算進結果。
```

這句才會影響下次決策。

它告訴 AI：

- 什麼時候要想起這條規則：金額加總或計數。
- 要檢查什麼：資料來源是否包含軟刪除或舊資料。
- 為什麼要檢查：避免錯把不應計入的資料算進結果。

後來我把 `qa-thought-capture` 設計成一個執行中的攔截器，就是為了抓這種句子。

當測試過程中出現一個非顯然、可複用、會影響未來決策的 QA 判斷時，先把它標成候選：

```text
採集候選：驗金額加總或計數前，先確認資料來源是否包含軟刪除或舊資料 → 建議落點 docs/QA_HEURISTICS.md
```

但它不會自動寫進 repo。

這一點很重要。

自動採集只負責「不要漏掉」。真正儲存前，還是要經過人類確認。因為如果讓 AI 自己決定什麼知識值得永久保存，它很快就會把 repo 寫滿看似有道理、其實沒有通過 QA 判斷的句子。

所以回收流程大概是：

```text
測試中冒出 QA 判斷
  ↓
AI 標記採集候選
  ↓
人類確認：存 / 改後存 / 拒絕 / 補充
  ↓
寫進正確落點
  ↓
下次 agent 在正確時機讀到
```

我也越來越重視穩定的 ID 或穩定的錨點。

因為沒有 ID，知識很容易膨脹。下次遇到相似情境，AI 可能不是更新同一條規則，而是又新增一條差不多的。久了以後，同一個概念散在三個地方，內容還不完全一致。

所以一條可維護的知識，最好有穩定位置：

- memory 裡用穩定 slug，例如 `feedback_*`。
- QA heuristic 放在明確分類底下。
- product note 放在對應模組文件。
- registry 放在對應頁面。
- seed 知識放在對應 catalog 或 helper 文件。

穩定位置的目的不是形式好看，而是讓未來能更新同一條，而不是一直追加新垃圾。

## 知識會爛掉：回收不只是加，還要修剪

這是我覺得最少人講的一段。

大家談 AI knowledge base，通常都在講怎麼加知識、怎麼寫文件、怎麼做 RAG、怎麼讓 AI 讀更多資料。

但用久了會發現，知識不是越多越好。

知識會爛掉。

有些規則是過時的。當時的 UI、API、產品行為、測試策略已經變了，但文件還留著。

有些規則是重複的。三個地方各寫了一版，語氣不同、範圍不同，AI 讀完不知道哪個才準。

有些規則太細。原本只是某張票的臨時情境，卻被寫成通用規則，之後反而誤導其他測試。

有些規則太長。它不是幫 AI 聚焦，而是在增加噪音。

這就是 context rot。

當知識堆久了，問題不再是「AI 沒有資料」，而是「AI 被太多品質不一的資料干擾」。規則檔越來越長，skill 越來越厚，prompt 越來越重，AI 反而越抓不到重點。

所以知識回收不能只有新增。

它還要有修剪。

我後來接受一個反直覺的想法：

> 一個健康的知識庫，會變小的次數應該跟變大的次數差不多。

至少它要允許變小。

這也是為什麼 `qa-thought-capture` 不只要能 append，還要能 supersede 和 delete。

當一條規則被新事實取代時，不應該在旁邊再加一條「補充說明」。如果新舊兩條同時存在，下一個 agent 很可能兩條都讀到，然後自己猜哪個比較新。

比較好的做法是直接取代舊規則，標清楚：

```text
[2026-06-24 取代原假設：舊規則 X]
新規則 Y。
```

如果一條規則被證明是錯的，而且沒有保留價值，就刪掉。

這件事做起來不刺激，也不會馬上看到成果。但長期來看，它決定知識庫是資產，還是負債。

## 不只記正確做法，也要記錯誤樣本

知識回收還有另一個常被忽略的面向：負面樣本。

大多數知識庫只記「應該怎麼做」。

但 AI 很常重犯的，不是它完全不知道正確做法，而是它在某些情境下又走回舊路。

所以我開始覺得，「上次錯在哪」也應該被回收。

正面規則教 AI 走對路。

負面樣本攔 AI 走回頭路。

例如，正面規則可以寫：

```text
狀態類驗收要走到前端 UI 呈現確認，不能只看 API 回 200。
```

負面樣本可以補上：

```text
不要把 write API 回 200 當成使用者看到的狀態已正確；這只能證明寫入請求成功，不能證明 UI 呈現正確。
```

兩者看起來很像，但效果不同。

前者告訴 AI 該做什麼。

後者讓 AI 認得一個常見錯誤：「我已經看到 API 成功，所以應該可以宣告驗收通過。」

在 AI QA 裡，負面樣本尤其重要，因為很多錯誤不是語法錯，而是判斷錯。AI 會做出一個看起來合理、其實 QA 不該接受的結論。

把這些錯誤樣本留下來，等於把人的審查直覺變成下一次 AI 的警示牌。

## 回收要有固定儀式

知識回收最難的地方不是知道要存哪。

而是持續做。

因為回收和修剪都屬於那種「今天不做也不會立刻死，但長期不做一定會爛」的工作。

你可以今天不寫 worklog，明天不整理 registry，後天不更新 product note。短期看起來完全沒事。直到某天換一個 agent 接手，它又問同樣問題、踩同樣坑、重做同樣探索，你才會發現前面省下的時間其實都變成後面的債。

所以我不想依賴意志力。

我需要固定儀式。

在 my-playwright 裡，這個儀式通常發生在幾個時機：

- 測試過程中，出現可複用 QA 判斷時先標成採集候選。
- 工作告一段落時，用 `/capture` 或收尾檢查把候選攤開。
- 人類逐條裁決：存、改後存、拒絕、補充。
- 寫回對應落點。
- 若發現既有規則過時，直接取代或刪除。

這些動作不需要每次都很重。

重點是它要固定。

如果知識回收只靠「想到再記」，它一定會失敗。因為真正有價值的 QA 直覺通常出現在忙、卡、趕、正在查問題的時候。那正是人最不會主動停下來寫文件的時候。

所以我把它變成流程的一部分，而不是額外的好習慣。

## 一條知識的生命週期

一條回收知識，不是寫下來就結束。

它應該有生命週期。

```text
某次測試中發現一個非顯然判斷
  ↓
標成採集候選
  ↓
人類確認後寫入正確落點
  ↓
下次 test plan / POM / seed 設計時被讀到
  ↓
實際影響 AI 的決策
  ↓
後來產品或流程改變
  ↓
這條知識被更新、合併或刪除
```

這個生命週期讓我重新定義「知識管理」。

它不是把文件寫多。

它是讓知識能進來、能被用到、能被修正、能被淘汰。

只有進沒有出，就不是知識管理，是堆積。

## 建立回收機制後，實際改變了什麼

最明顯的改變，是我不用一直重新教同一件事。

以前 AI 常常問一些我已經回答過很多次的問題，或重新踩以前修過的 selector、seed、report 證據問題。現在它至少有機會在開始前讀到既有規則，知道哪些地方以前踩過、哪些判斷已經沉澱。

第二個改變，是 AI 的產出比較有累積感。

不是每次都像一個全新的外包人員從零開始，而是比較像一個會讀交接文件、會查既有資產、會延續前面決策的協作者。

第三個改變，是我比較能分辨「這是一次性修正」還是「這是值得回收的 QA 判斷」。

這個分辨本身很重要。不是所有東西都值得留下來。太多一次性細節進入知識庫，未來只會造成噪音。

當然，它也有代價。

回收要花時間，修剪也要花時間。尤其修剪很難，因為刪除知識比新增知識更需要判斷：這條是真的過時了，還是只是我今天沒用到？這條該刪，還是該改寫？這條是通用規則，還是只屬於某張票？

這些問題目前不可能完全自動化。

但我寧願保留這個人工關卡。

因為 AI QA 的知識庫如果完全自動長大，很快就會變成另一個需要 QA 的東西。

## 如果重來會怎麼做

如果回到四個月前，我會更早建立知識回收機制，而且會做得更輕。

我不會一開始就設計很完整的知識架構。

我會先做三件事：

1. 定義哪些校正值得回收。
2. 定義它們要流到哪幾個固定落點。
3. 定義什麼時候要修剪或取代舊規則。

這樣就夠開始了。

很多人導入 AI QA，第一件事會想自動化更多測試。但我現在更相信，前期更重要的是建立兩個基礎機制：

第一，需求理解機制：讓 AI 在動手前先把理解攤開，讓人可以校正。

第二，知識回收機制：讓校正之後的知識流回 repo，下次不需要重講。

沒有第一個機制，AI 會很有自信地把錯誤需求做完。

沒有第二個機制，AI 會一直重犯你已經教過它的錯。

所以我給想導入 AI QA 的人最實際的建議是：

不要只顧著教 AI。

先想好教完的東西去哪裡。

也要想好，什麼時候該刪。
